# Phase 4.5 — Power resilience with NUT (laptop battery as UPS)

No UPS purchased. The cluster nodes are laptops, so their **batteries** are the
UPS. Network UPS Tools (NUT) reads the laptop battery and, on mains loss,
gracefully shuts down the battery-less devices (NAS, Pi) with voice alerts via
the node speakers.

> **The trick:** NUT expects real UPS hardware. A laptop battery isn't one. A
> bridge script reads the kernel battery/AC state and feeds it into NUT's
> `dummy-ups` driver, so the battery *looks like* a UPS to the whole NUT network.

**Topology:** pve-01 + pve-02 = NUT servers (each its own battery). nas-01 +
qdev-01 = NUT clients (shut down on Low Battery).

---

## Battery device names (Dell Precision 3561)

```
/sys/class/power_supply/BAT0/capacity   # % charge
/sys/class/power_supply/BAT0/status     # Full/Charging/Discharging
/sys/class/power_supply/AC/online       # 1 = mains, 0 = battery
```

> Note: it's `AC` (not `AC0`/`ADP1`). Batteries are worn (~55–66% of design
> capacity) but irrelevant — even ~50 Wh gives hours of runtime vs the ~5 min
> needed for graceful shutdown.

## Stage 2 — Alert scripts (both nodes, `/usr/local/sbin/`)

```bash
# homelab-alert-warning.sh  (50% / on-battery)
#!/bin/bash
NODE=$(hostname -s); BAT=$(cat /sys/class/power_supply/BAT0/capacity)
logger -t homelab-power "WARNING: $NODE on battery, level ${BAT}%"
espeak "Warning. ${NODE} is running on battery power. Level ${BAT} percent."

# homelab-alert-critical.sh  (30%)
#!/bin/bash
NODE=$(hostname -s); BAT=$(cat /sys/class/power_supply/BAT0/capacity)
logger -t homelab-power "CRITICAL: $NODE battery ${BAT}%, shutdown approaching"
espeak "Caution. ${NODE} battery critical at ${BAT} percent. Prepare for shutdown."

# homelab-alert-shutdown.sh  (20%)
#!/bin/bash
NODE=$(hostname -s)
logger -t homelab-power "SHUTDOWN: $NODE initiating graceful shutdown sequence"
for i in 1 2 3; do espeak "Shutdown imminent on ${NODE}. Cluster powering down."; sleep 2; done

# homelab-alert-recovery.sh  (mains restored)
#!/bin/bash
NODE=$(hostname -s)
logger -t homelab-power "RECOVERY: $NODE mains power restored"
espeak "Power restored on ${NODE}. Shutdown cancelled. All clear."

# homelab-shutdown.sh  (the REAL shutdown command upsmon calls)
#!/bin/bash
NODE=$(hostname -s)
logger -t homelab-power "SHUTDOWN initiated on ${NODE}"
/usr/local/sbin/homelab-alert-shutdown.sh
for vmid in $(qm list | awk 'NR>1 {print $1}'); do qm shutdown "$vmid" --timeout 60 & done
wait
/sbin/shutdown -h +0

# nut-notify.sh  (dispatcher — upsmon calls this with $NOTIFYTYPE)
#!/bin/bash
NODE=$(hostname -s)
logger -t homelab-power "NUT event: ${NOTIFYTYPE} on ${NODE}"
case "$NOTIFYTYPE" in
  ONBATT)   /usr/local/sbin/homelab-alert-warning.sh ;;
  LOWBATT)  /usr/local/sbin/homelab-alert-critical.sh ;;
  ONLINE)   /usr/local/sbin/homelab-alert-recovery.sh ;;
  SHUTDOWN) /usr/local/sbin/homelab-alert-shutdown.sh ;;
esac
```
`chmod +x /usr/local/sbin/homelab-* /usr/local/sbin/nut-notify.sh`

> **Why node speakers, not the Pi:** the laptops *are* the battery source — one
> hop from sensor to alert. Pi stays single-purpose. Two laptops = redundant audio
> in two rooms, free.

> **Why local audio for late alerts:** a 50% alert can use push/email while
> internet's up, but at 20% during a real outage the modem is already dead. Any
> alert that must fire *during* the event has to be local. Never let the alert
> path depend on the thing that's failing.

## Stage 3 — The battery→NUT bridge (atomic write)

`/usr/local/sbin/nut-battery-bridge.sh`:
```bash
#!/bin/bash
STATEFILE="/etc/nut/laptopups.seq"
TMPFILE="/etc/nut/.laptopups.seq.tmp"
BAT_PATH="/sys/class/power_supply/BAT0"
AC_PATH="/sys/class/power_supply/AC"
LOW_THRESHOLD=20
WARN_THRESHOLD=30

charge=$(cat "${BAT_PATH}/capacity" 2>/dev/null || echo 100)
ac_online=$(cat "${AC_PATH}/online" 2>/dev/null || echo 1)

if [ "$ac_online" = "1" ]; then
    status="OL"
else
    if [ "$charge" -le "$LOW_THRESHOLD" ]; then status="OB LB"; else status="OB"; fi
fi

cat > "${TMPFILE}" <<INNER
device.mfr: Dell
device.model: Precision3561-BAT0
device.type: ups
battery.charge: ${charge}
battery.charge.low: ${LOW_THRESHOLD}
battery.charge.warning: ${WARN_THRESHOLD}
ups.status: ${status}
ups.mfr: Dell
ups.model: Laptop Battery UPS
INNER

chown root:nut "${TMPFILE}"; chmod 660 "${TMPFILE}"
mv -f "${TMPFILE}" "${STATEFILE}"     # atomic — driver never sees a half-written file
```

> **Status mapping:** mains = `OL` (on line), battery = `OB` (on battery), battery
> below 20% = `OB LB` (low battery — this triggers shutdown).

> **Why atomic write (temp + mv):** writing the file in place while the driver
> reads it causes "driver not connected" crashes. Writing a temp file then `mv`
> (atomic rename) means the driver always sees a complete file.

## Stage 4 — NUT config

```bash
apt install -y nut nut-server nut-client
```

Define passwords **once** as shell variables so the two files can't drift, then
write configs with an **unquoted** heredoc so the variables expand:

```bash
MASTERPASS="<MASTER_PASS>"   # real value in Vaultwarden; alphanumeric only
CLIENTPASS="<CLIENT_PASS>"

# /etc/nut/ups.conf  — loop mode, .seq file (NOT .dev)
cat > /etc/nut/ups.conf <<'EOF'
[laptopups]
    driver = dummy-ups
    port = laptopups.seq
    desc = "pve-01 laptop battery as UPS"
    mode = dummy-loop
EOF

# /etc/nut/nut.conf
echo "MODE=netserver" > /etc/nut/nut.conf

# /etc/nut/upsd.conf  — listen on localhost + this node's LAN IP
cat > /etc/nut/upsd.conf <<'EOF'
LISTEN 127.0.0.1 3493
LISTEN 192.168.1.100 3493
EOF

# /etc/nut/upsd.users  — unquoted EOF so ${VARS} expand
cat > /etc/nut/upsd.users <<EOF
[upsmon]
    password = ${MASTERPASS}
    upsmon master

[slaveclient]
    password = ${CLIENTPASS}
    upsmon slave
EOF

# /etc/nut/upsmon.conf  — MASTERPASS must match upsd.users [upsmon]
cat > /etc/nut/upsmon.conf <<EOF
MONITOR laptopups@localhost 1 upsmon ${MASTERPASS} master
MINSUPPLIES 1
SHUTDOWNCMD "/usr/local/sbin/homelab-shutdown.sh"
POLLFREQ 5
POLLFREQALERT 5
HOSTSYNC 15
DEADTIME 15
NOTIFYCMD /usr/local/sbin/nut-notify.sh
NOTIFYFLAG ONLINE   SYSLOG+EXEC
NOTIFYFLAG ONBATT   SYSLOG+EXEC
NOTIFYFLAG LOWBATT  SYSLOG+EXEC
NOTIFYFLAG SHUTDOWN SYSLOG+EXEC
EOF

chown -R root:nut /etc/nut/
chmod 640 /etc/nut/*.conf /etc/nut/upsd.users
```

> **`dummy-loop` + `.seq`, not `dummy-once` + `.dev`:** dummy-ups only re-reads
> live changes in loop mode, which it selects when the state file ends in `.seq`.
> With `.dev` it reads once at startup and the "UPS" stays frozen at 100% forever.

> **No `POWERDOWNFLAG /etc/killpower`:** I dropped this line — it contributed to
> FSD latching during testing and isn't needed for a battery-bridge UPS.

## Stage 5 — Bridge timer + start order

```bash
# systemd timer to refresh the state file every 10s
cat > /etc/systemd/system/nut-battery-bridge.service <<'EOF'
[Unit]
Description=Update NUT dummy-ups from laptop battery
After=multi-user.target
[Service]
Type=oneshot
ExecStart=/usr/local/sbin/nut-battery-bridge.sh
EOF

cat > /etc/systemd/system/nut-battery-bridge.timer <<'EOF'
[Unit]
Description=Run NUT battery bridge every 10s
[Timer]
OnBootSec=30
OnUnitActiveSec=10s
AccuracySec=1s
[Install]
WantedBy=timers.target
EOF
systemctl daemon-reload

# Start in the ONLY order that works: state → driver → server → monitor
/usr/local/sbin/nut-battery-bridge.sh
chown root:nut /etc/nut/laptopups.seq ; chmod 660 /etc/nut/laptopups.seq
upsdrvctl stop 2>/dev/null ; pkill -9 -f dummy-ups 2>/dev/null
sleep 2 ; rm -f /run/nut/*.pid
upsdrvctl start ; sleep 3
systemctl start nut-server ; sleep 2
systemctl start nut-monitor ; sleep 3
systemctl enable nut-server nut-monitor nut-battery-bridge.timer
systemctl start nut-battery-bridge.timer

# Verify
upsc laptopups@localhost | grep -E 'battery.charge|ups.status'   # → 100 / OL
journalctl -u nut-monitor --no-pager | tail   # → NO "ACCESS-DENIED"
```

## Stage 6 — Safe shutdown dry-run

Never dry-run a real `shutdown -h` on a live cluster. Swap in a logging-only
stand-in, simulate low battery, confirm it *would* fire, then restore.

```bash
cp /usr/local/sbin/homelab-shutdown.sh /usr/local/sbin/homelab-shutdown.sh.real
cat > /usr/local/sbin/homelab-shutdown.sh <<'EOF'
#!/bin/bash
NODE=$(hostname -s)
logger -t homelab-power "DRY-RUN: shutdown would trigger now on ${NODE}"
espeak "Dry run. ${NODE} would shut down now. This is only a test."
EOF
chmod +x /usr/local/sbin/homelab-shutdown.sh

systemctl stop nut-battery-bridge.timer   # stop it racing the test
# write a 15% / OB LB state into /etc/nut/laptopups.seq, wait 20s
journalctl -t homelab-power | tail   # expect ONBATT→LOWBATT→SHUTDOWN→DRY-RUN

# Restore + verify
cat > /usr/local/sbin/homelab-shutdown.sh <<'EOF'
#!/bin/bash
NODE=$(hostname -s)
logger -t homelab-power "SHUTDOWN initiated on ${NODE}"
/usr/local/sbin/homelab-alert-shutdown.sh
for vmid in $(qm list | awk 'NR>1 {print $1}'); do qm shutdown "$vmid" --timeout 60 & done
wait
/sbin/shutdown -h +0
EOF
chmod +x /usr/local/sbin/homelab-shutdown.sh
grep -E 'shutdown -h|DRY-RUN' /usr/local/sbin/homelab-shutdown.sh  # MUST show real, NO dry-run
/usr/local/sbin/nut-battery-bridge.sh
systemctl start nut-battery-bridge.timer
```

## 💥 Lessons — NUT is gloriously finicky

1. **Background timer raced the driver.** The 10s bridge timer collided with the
   driver re-reading → "duplicate driver" / "driver not connected." Fix: stop the
   timer during manual tests; atomic writes in the bridge.
2. **`dummy-once` vs `dummy-loop`.** Frozen at 100% until I switched to loop mode
   + `.seq` file.
3. **FSD latches after a dry-run.** `ups.status: FSD OL` stuck because the system
   never actually powered off. Real outages clear it on next boot; test runs need
   a full stack restart (`pkill upsmon upsd dummy-ups`, `rm /run/nut/*.pid`,
   `rm -f /etc/killpower`, restart in order). Also caught my "restore real script"
   step had **silently failed** — verified instead of assuming, found the dry-run
   version still in place.
4. **Password mismatch = silent half-failure.** `upsmon` read status fine with a
   wrong password but privileged actions got `ERR ACCESS-DENIED` — detection
   looked healthy while shutdown would never have fired. Fix: define each password
   once as a shell variable, write both files from it (single source of truth).
5. **NUT passwords: alphanumeric only.** `/` and `+` from `openssl rand -base64`
   can break config parsing — use `openssl rand -hex` instead.

The dry-run is precisely what surfaced all of these *safely* instead of at 3am
during a real cut.
